# ADR R17: снятие `company.head_company_flag` — очистка дочерних UF и ИБ 34

## Статус

Принято.

## Контекст

UF **`UF_CRM_1758028816`** (`company.holding`) ссылается на **элемент инфоблока 34**; у элемента свойство **`B24_COMPANY_ID`** = CRM ID компании. При снятии галочки **`UF_CRM_1758028888`** (`company.head_company_flag`) у головной компании нужно:

1. **Деактивировать** соответствующий элемент в ИБ **34** (поиск по `B24_COMPANY_ID` = ID головной компании).
2. У **всех дочерних** компаний, у которых в **`UF_CRM_1758028816`** указана привязка к этой головной (ID элемента ИБ, CRM ID или legacy `CO_*`), **очистить** **`UF_CRM_1758028816`** и **`UF_CRM_1776426878`** (`company.holding_group_members`).

## Решение

В **`CompanySync::onBeforeCompanyUpdate`** после успешной политики: если по снимку UF было «головная портала», а в `$arFields` флаг снимается — вызывается **`clearHoldingChildrenWhenPortalHeadFlagRemoved`**.

В исходящей синхронизации **`HOLDING_IBLOCK_ID`** и **`HOLDING_UF_IBLOCK_ID`** совпадают (**34**) — отдельного ИБ 57 в конфигурации нет; каталог холдингов один.

## Риски

- Если CRM не передаёт ключ UF в `$arFields` при снятии галочки, переход не детектируется — очистка не выполнится.
- Очистка UF у дочерних через `false` зависит от типов полей в CRM.
- Исторически в коде фигурировало два ИБ; на актуальном портале — один каталог (**ИБ 34**).
