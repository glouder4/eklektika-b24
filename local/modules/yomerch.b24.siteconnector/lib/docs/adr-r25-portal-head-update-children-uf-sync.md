# ADR R25: обновление головной по порталу — дочерние: скидка и UF участников холдинга

## Статус

Принято.

## Контекст

Для компании с **`UF_CRM_1758028888`** (`company.head_company_flag`, головная по холдингу на портале) при любом сохранении карточки нужно:

1. Найти элемент **ИБ 34**, у которого **`B24_COMPANY_ID`** = CRM ID этой компании.
2. Найти дочерние компании с **`UF_CRM_1758028816`** (`company.holding`), равным **ID элемента** (или согласованным legacy-значением — см. `getChildB24IdsLinkedToHoldingIblock`).
3. У дочерних выровнять **`UF_CRM_1771490556028`** (`company.discount`) со значением головной.
4. На головной и всех дочерних выставить одинаковый **`UF_CRM_1776426878`** (`company.holding_group_members`): список CRM ID головной и всех дочерних.

Ранее скидка с головы к дочерним пробрасывалась только при **`company.is_head`** и наличии ключа скидки в `$arFields`; UF участников холдинга пересчитывался только если в `$arFields` явно пришли ключи портальной «головы» или холдинга — при частичном сохранении головной карточки дочерние не обновлялись. При сохранении **дочерней** без смены UF холдинга в запросе список **`UF_CRM_1776426878`** на головной и соседних дочерних не пересчитывался.

## Решение

В **`CompanySync::onAfterCompanyUpdate`**:

- По снимку и `$arFields` определяется **`isHeadPortalCompany`**; при истине вызывается **`ensureHoldingUfCatalogElementForPortalHeadCompany`** до кластерных операций (чтобы запись ИБ 34 существовала).
- **`syncHoldingGroupMembersAcrossCluster`** дополнительно вызывается при каждом сохранении компании с включённым портальным флагом головной (не только при наличии соответствующих ключей в `$arFields`).
- Тот же пересчёт **`UF_CRM_1776426878`** вызывается при сохранении **дочерней**: если по снимку резолвится элемент кластера (**`resolveHoldingIblockElementForOutboundCluster` > 0**), при этом нет портального флага головной и нет **`company.is_head`** — чтобы при любом изменении карточки дочерней головная и все участники получили актуальный список ID.
- Резолв элемента ИБ 34 для кластера: не только **`company.is_head`**, но и **`company.head_company_flag`** — иначе у портальной головной без флага «головная» в старом смысле и без заполненного UF холдинга на самой себе **`resolveHoldingIblockElementForOutboundCluster`** возвращал 0, в **`UF_CRM_1776426878`** попадал только ID головной компании (ветка fallback без дочерних). Аналогично **`resolveHoldingCompanyB24Ids`**.
- Проброс скидки к дочерним для портальной головной: текущее значение скидки из снимка компании и **`propagateHeadDiscountToChildrenAndContacts`**; ветка **`company.is_head`** + ключ скидки в `$arFields` сохранена для прежнего сценария ИБ 16.

## Почему эта логика в `yomerch.b24.outbound`, а не в «чисто внутреннем» модуле

- **`CompanySync`** изначально — оркестратор **контура CRM → сайт**: класс в пространстве имён **`OnlineService\Sync\ToSite`**, наследует **`OutboundRequest`** и в том же **`onAfterCompanyUpdate`** после внутренних правок собирает **`UPDATE_COMPANY`** и дергает **`sendRequest`** (см. регистрацию обработчиков в **`yomerch.b24.siteconnector`** → `register_bitrix_handlers.php`).
- События Bitrix повешены на этот класс не потому, что UF холдинга «внешние», а потому что **одно сохранение компании** должно в одном предсказуемом порядке: политики/UF/ИБ → затем исходящий payload и побочные CRM-обновления (контакты, `markInboundCompanyUpdate` и т.д.). Вынести только «внутреннюю» часть в другой модуль можно, но тогда нужен явный слой координации и общие флаги — иначе легко получить рассинхрон с сайтом или двойной outbound.
- Название модуля **outbound** отражает **главную ответственность** пакета, а не границу «только HTTP наружу». Смешение в одном классе — компромисс ясности структуры vs связность одного пайплайна.

## Риски

- Дополнительные записи в CRM при каждом сохранении головной компании (обновление дочерних UF и контактов по скидке).
- Дополнительные записи при каждом сохранении дочерней в холдинге (полный пересчёт **`UF_CRM_1776426878`** по кластеру).
- Рекурсивные срабатывания `onAfter` при массовом обновлении UF участников — как и раньше, зависят от состава `$arFields` в событии Bitrix.
