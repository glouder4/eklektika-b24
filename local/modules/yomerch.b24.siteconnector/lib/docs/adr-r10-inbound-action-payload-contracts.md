# ADR: R10 inbound ACTION payload contracts (сайт → CRM)

- Status: accepted
- Date: 2026-04-30
- Short id: **R10**
- Scope: явные предметные контракты тел запросов для каждого `ACTION`, поддерживаемого `InboundEndpoint`, и выравнивание поиска/обновления компании по UF «элемент сайта» с каноническим и legacy-полем

## Context

Внешний транспортный контракт (Tier A/B, R7/R8) описывает вход, безопасность и наблюдаемость, но не заменяет спецификацию обязательных полей для каждого `ACTION`. Без этого интеграции могут заполнять только legacy UF (`UF_CRM_3804624439373`), тогда как поиск входящего `UPDATE_COMPANY` опирался только на канонический UF (`UF_CRM_1774915439581`), что давало расхождения и «пустой»/`0` идентификатор элемента на сайте при ошибочном порядке операций.

## Decision

1. Ввести отдельную папку **`local/bitrix24-inbound-from-site-contracts/`** с README и одним файлом на каждый поддерживаемый `ACTION` (плюс описание алиаса `UPDATE_STATUS_GROUP`).
2. Зафиксировать в коде **паритет чтения** для идентификатора элемента сайта: канонический UF, legacy UF или `SITE_ELEMENT_ID`; поиск компании по обоим UF; при явном обновлении UF — запись **обоих** полей тем же значением (как при исходящем `CompanySync`).
3. Отклонять строковые значения **`''`** и **`'0'`** как недопустимый идентификатор элемента для `UPDATE_COMPANY`.

## Consequences

- Интеграторам проще согласовать ожидания с кодом без чтения всего модуля inbound.
- Legacy-сценарии продолжают работать при условии корректного непустого ID элемента.
- Дублирующие записи с одинаковым ID в двух UF у одной компании безопасны для поиска; конфликты между разными компаниями по одному значению по-прежнему дают ошибку неуникальности.

## Risks

- Жёсткий отказ от `'0'` может затронуть тестовые заглушки — их нужно перевести на валидные тестовые ID или не использовать `UPDATE_COMPANY` без реального элемента.
- Документы в `bitrix24-inbound-from-site-contracts` нужно обновлять при добавлении новых ACTION в диспетчер.
