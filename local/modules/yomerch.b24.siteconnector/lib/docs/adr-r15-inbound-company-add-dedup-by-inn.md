# ADR R15: входящий `crm.company.add` при совпадающем ИНН (холдинг)

## Статус

Заменено уточнением **2026-04-30** (дочерняя компания под головной, см. текст ниже).

## Контекст

Повторная регистрация с **уже известным ИНН** в CRM не должна порождать произвольную вторую «равную» компанию без правил холдинга.

## Решение (актуально)

1. По **`RQ_INN`** ищется компания в реквизитах.
2. **Одна** компания и UF **`UF_CRM_1758028888`** (`company.head_company_flag`) истинно → выполняется **`CCrmCompany::Add`** для **новой** записи; у дочерней в **`UF_CRM_1758028816`** (`company.holding`) указывается **CRM ID головной** компании.
3. Формируется полный список участников (головная + дочерние по UF холдинга + уже сохранённый список в **`UF_CRM_1776426878`**), затем **`UF_CRM_1776426878`** (`company.holding_group_members`) **одинаково** обновляется на всех компаниях этого набора (формат `CO_<ID>`, fallback — числовые ID).
4. Если ИНН найден, но головной признак **не** включён — **без** создания компании: возвращается ID существующей компании и `company_add_use_existing_for_contact` (сценарий только привязки контакта).
5. Несколько компаний с одним ИНН — `company_add_ambiguous_inn`.

## Риски

- Несовпадение формата ИНН в реквизите и в запросе — отдельная ветка не сработает.
- Тип UF «Фирмы холдинга» может требовать иной формат значений — при сбое смотреть второй проход с числовым списком в коде.
