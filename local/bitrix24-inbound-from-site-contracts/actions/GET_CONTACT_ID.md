# ACTION: `GET_CONTACT_ID`

Поиск контакта CRM **только по email** (регистрация дилера с сайта).

## Авторизация (транспорт)

Без валидного токена запрос **не дойдёт** до логики поиска: портал вернёт **`403`** `sync_forbidden`, в логе будет `reason_detail: missing_or_empty_token`.

Обязательно передайте **`X-SYNC-TOKEN`** (значение = `sync_token` с портала) **или** поле **`sync_token`** / **`SYNC_TOKEN`** / **`_TOKEN`** / **`_SYNC_TOKEN`** в том же теле, что и `ACTION` / `EMAIL`.

## Обязательные поля тела запроса

| Поле | Тип | Описание |
|------|-----|----------|
| `ACTION` | string | `GET_CONTACT_ID` |
| `EMAIL` | string | Email контакта (после trim непустой) |

## Идентификация контакта

| Поле | Использование |
|------|----------------|
| `EMAIL` | **Единственный ключ** поиска. Ровно один контакт с этим email → успех. |
| `PHONE` | **Игнорируется** для поиска и для решения «контакт уже существует». Может присутствовать в запросе для совместимости — на логику не влияет. |

Порядок в коде (`InboundEndpoint::handleGetContactId`):

1. `EMAIL` обязателен; пустой → `invalid_payload` (HTTP 400).
2. Поиск по точному совпадению email в CRM.
3. Ровно **один** контакт → `success: 1`.
4. Ноль или несколько контактов → `contact_not_found` / `contact_lookup_ambiguous`.

### Регистрация на сайте

- Уникальность контакта — **только email**.
- Один и тот же телефон может быть у нескольких контактов с разными email.
- Сайт шлёт `GET_CONTACT_ID` **без** `PHONE` (или с `PHONE` — CRM его не использует для lookup).

## Успешный ответ

- `success`: `1`
- `data.ID`: числовой ID контакта в CRM
- `data.UF_CRM_3804624445748` (если заполнен): `contact.delete_site_ref` — ref на сайте
- `data.EMAIL`, `data.PHONE` (опционально): данные контакта **для справки**, не для проверки уникальности

## Ошибки

| Ситуация | `reason_code` |
|----------|----------------|
| Нет `EMAIL` | `invalid_payload` |
| Не найден | `contact_not_found` |
| Несколько контактов с одним email | `contact_lookup_ambiguous` |
